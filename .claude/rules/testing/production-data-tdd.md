# 変則TDD（本番データ駆動開発）

## 原則

> モックデータでテストすると本番でコケる。
> 本番データの「汚さ」はモックで再現不可能。

## 層別テスト戦略

| 層 | テストデータ | 理由 |
|----|------------|------|
| Adapter | **本番データコピー** | 実際の「汚さ」を検証 |
| Core | モックTDD可 | 標準形式前提で純粋 |
| 結合 | 本番コピー → 段階的展開 | 本番互換性検証 |

## 本番データの「汚さ」例

モックで再現不可能なパターン：

- 結合セル
- 空白行/列
- 全角/半角混在
- 日付形式バラバラ（2024/1/1, 2024-01-01, R6.1.1）
- 数値に見える文字列
- 予期しないヘッダー

## Adapter層のテストフロー

```
1. 本番データをコピー（匿名化が必要なら実施）
2. Adapterに入力
3. 出力が標準形式に準拠することを検証
4. エッジケースは本番データから抽出
```

## Core層のテスト

- 標準形式のモックデータでTDD可
- Adapter層で入力品質が保証されている前提
- 従来のテスト手法を適用

## 段階的展開

```
1件 → 10件 → 100件 → 全件
```

| 段階 | 目的 |
|------|------|
| 1件 | 基本動作確認 |
| 10件 | 主要パターン網羅 |
| 100件 | エッジケース発見 |
| 全件 | 本番投入 |

## テスト設計の優先順位

1. **Adapter層を本番データで徹底テスト**
2. Core層は標準形式前提でシンプルにテスト
3. 結合テストは段階的に拡大

## ⛔ 禁止事項（テストデータ品質）

本番データ駆動の原則に違反するテストは `npm run test:validate-data` で自動検出される。

| 禁止 | VTD Rule | 理由 |
|------|----------|------|
| 空配列でテスト（`headers: []`） | VTD-001 | 本番データの汚さを検証していない |
| 全値が空/ゼロのオブジェクト | VTD-002 | ダミーデータは本番を再現しない |
| `toBeDefined()` のみ | VTD-003 | 値の正しさを検証していない |
| `expect(true).toBe(true)` | VTD-004 | 何も検証していない |
| モックデータでのフォールバック | - | エラーを隠蔽する |

詳細: `.claude/rules/testing/testing-standards.md`
