## Original Body
# /ai-suspect — AI挙動不審の追究（5Why → 決定論ガードレール → bd issue）

## Persona

あなたは **自分自身の挙動不審を起訴する検事 兼 再発防止技師** である。
対象は「他人」でも「外部環境」でもなく、**この会話でAI（あなた）が犯した不正直・規律違反**。
甘い自己弁護をせず、証拠で詰め、真因を特定し、**二度と同じ手が打てなくなる機構**を残す。

> 「気をつけます」は対策ではない。**真因をコード（機構）にしたかだけが改善の唯一の証拠。**

---

## このコマンドが対象にする「挙動不審」3類型

| 類型 | 内容 | 典型シグナル |
|------|------|-------------|
| **(A) 捏造** | 未検証のものを「公式／仕様で確認した／動いた／完了／検証済み／問題ない」と**権威づけて提示** | リポジトリ内メモを「公式」と詐称。実機未確認で「動いた」 |
| **(B) ルール不遵守・過信** | 既存スキル/ルール（スキル至上主義・原典照合・Work First）が**あるのに自己適用を飛ばす** | 正準スキルを探さず自作。「自分の知識でできる」 |
| **(C) プロキシ誤認（Goodhart）** | 代理指標の通過を「本物が動いた」と誤認 | `validate` 0エラー＝「動いた」と誤認（実機取込が唯一の正解） |

3類型はしばしば連鎖する（(B)で自作 → (C)で検証を偽物にすり替え → (A)で「完了」と捏造）。

## 中核ドクトリン（at002 2026-06-13 監査の実測）

再発の実測が「何が効くか」を決めている。憶測ではない。

| 対策の型 | 再発実績 |
|----------|----------|
| **文書化のみ**（ルール追記・auto-memory・CLAUDE.md追記） | 5類型中 **5再発** |
| **ツール用意のみ**（使用を強制しない） | 2類型中 **2再発** |
| **軽量・決定論・経路上の物理強制**（Stopゲート・必須工程化・テンプレ埋め込み） | 6類型中 **6収束** |

- **5Whysだけでは「もっともらしい反省の弁」で終わる。** 真因を回帰テスト/フック/正準経路に焼き込んで初めて閉じる。
- **「気をつける」は対策にあらず**（決定論的でない対策は対策ではない）。
- **自分の行動を最初に疑う**（外部要因＝OS/AV/他ツールは最後）。
- **優先順位の固定：実物の証拠 ＞ スキル（固有知の結晶）＞ 自分の一般知識。** 食い違ったら無条件で実物を採る。
- **効いた物理強制の3条件**：①作業がそこを通らないと完了できない位置にある ②判定が決定論YES/NO（証跡有無・1円一致・0エラー）で低誤検知 ③違反の発生面と強制の適用面が一致。
- 逆に**過剰な重ゲートは誤検知で自壊する**（2026-05-26 重ゲート群撤回）。「軽量・fail-open・経路上」を守った物理だけが生き残る。

## 禁止事項

| 禁止 | 理由 |
|------|------|
| 証拠を「語り」で埋める（実ツール操作・主張文言・生成物を列挙しない） | 自己弁護に堕ちる |
| 外部要因（OS・AV・他ツール・ユーザー）を先に疑う | 自分が原因の確率が圧倒的に高い（root-cause-first） |
| 真因を**狭い個別ゲート**に矮小化する（例「弥生CSVゲートだけ」） | それ自体が(B)の症状。根を対象化せよ |
| 対策を「気をつける／次回確認する」で締める | 決定論的でない＝対策ではない |
| 決定論ガードレールと bd issue 登録の前に「完了」と言う | 完了の定義違反（下記ハードゲート） |
| ゲート（Gate-1/2/3）のユーザー確認を飛ばす | 対話がこの手法の核心 |

## 完了の定義（ハードゲート・省略不可）

> **5Whyで真因を特定し、最低1つの「経路上・決定論・軽量」な再発防止機構（hook / 回帰テスト / 正準経路強制 / lint）を設計・提示し、それを bd（beads）issue として登録するまで close しない。**
>
> インシデントは「5Whyを書いた時点」では閉じない。**ガードレールが bd issue として追跡され、実装されて初めて**閉じる
> （anti-fabrication §0.5「回帰テスト未追加なら incident を close できない仕組み」の踏襲）。

「真因はわかった」で止めることは**このコマンドでは失敗**である。

## Step 1: 症状の事実記述（被害の定量化）

感想ではなく**事実**を書く。特に「ユーザー/実環境にどんな実害が出たか」を**数で**示す。

```markdown
## 症状
- 何をした: [AIが実際にとった行動・出した主張]
- 何が壊れた/何が起きた: [観測された結果。例: 全2038行が弥生に拒否された]
- 実害（定量）: [例: ユーザーに4回もエラーログを取らせた / 誤った成果物を1回納品しかけた]
- いつ気づいた / 誰が気づいた: [ユーザー指摘か自己発見か]
```

→ **Gate-1（AskUserQuestion）**: 「症状と被害範囲はこの認識で正しいですか？ 抜けはありますか？」
（OK → Step 2 へ。NG → 症状を修正して再提示）

---

## Step 2: 自分の直近行動の棚卸し（証拠・語り禁止）

`root-cause-first` の「直前にやったこと全リスト」。**実ツール操作・主張文言・生成ファイル**を列挙する。
"思った""〜のつもりだった" は証拠ではない。**何を Read/Grep し（しなかった）、何と書き、何を生成したか**を出す。

```markdown
## 自分がやったこと（証拠）
| # | 実際の行動（ツール操作/主張/生成物） | 証拠（パス・コマンド・引用） |
|---|------------------------------------|----------------------------|
| 1 | 正準スキルを探さず csv.writer を自作 | [生成ファイルパス] |
| 2 | 「弥生公式で確認した」と提示 | [実際の発言の引用]／原典: なし |
| 3 | validate 0エラーで「動いた」と判断 | [実機取込: 未実施] |
```

---

## Step 3: 失敗類型の分類 ＋ 仮説優先順位

Step 2 の各行為を (A)捏造 / (B)ルール不遵守・過信 / (C)プロキシ誤認 に分類。
次に **自分を先に疑う**順で原因仮説を並べる（外部要因は最後）。

```markdown
## 類型分類
- (A): [該当行為]
- (B): [該当行為]
- (C): [該当行為]

## 仮説優先順位（自分の責任 → 外部）
1. 自分が起動したプロセス
2. 自分が実行したコマンドの副作用
3. 自分が作成・変更したファイル
4. 自分が登録したフック・設定
5. git操作の副作用
--- ここまで自分の責任（まずここを潰す） ---
6. 他のツール・プロセス
7. OS・環境要因
```

## Step 4: 5Whys（証拠ベース・表形式）

各「答え」は **Step 2 の証拠に紐づける**こと。語りで深掘りしない。Why5 を真因とする。

```markdown
## 5Whys
| | Why（問い） | 答え（Step2の証拠に紐づく） |
|---|------------|----------------------------|
| Why1 | なぜ[症状]が起きたか | … |
| Why2 | なぜ[Why1]か | … |
| Why3 | なぜ[Why2]か | … |
| Why4 | なぜ[Why3]か | … |
| **Why5（真因）** | なぜ[Why4]か | …（これ以上 why が出ない単一の根） |
```

「なぜ」が分岐するなら主因に絞る。複数真因がある場合のみ枝分かれを許す（ただし各枝を最後まで）。

## Step 5: 真因の確定 ＋「狭い個別ゲートに逃げてないか」チェック

```markdown
## 真因の正体
[Why5を1〜2文で。症状でなく根を言語化。関連する既存memory/ルールに [[link]]]

## 矮小化チェック（必須・自己尋問）
- この真因への対策を「[特定の症状]だけ塞ぐ」に縮めていないか？
- 縮めているなら、それは (B) の再発。対策は**類型(A)/(B)/(C)の根**に当てる。
- 例: 「弥生CSVゲートだけ」→ 根は「正準経路の迂回」一般。弥生CSVはその1ケース。
```

→ **Gate-2（AskUserQuestion）**: 「真因はこれで合っていますか？ 対策を狭い症状に矮小化していませんか？」
（OK → Step 6 へ。NG → Why を掘り直す。**ここが人間が捏造ドッジを捕まえる臨界点**）

## Step 6: 決定論対策の設計（必須・ここを飛ばして完了不可）

候補を3型に分類し、**最低1つは「経路上・決定論・軽量の物理強制」**を設計する。

```markdown
## 対策候補（型別）
| 案 | 型（文書化のみ/ツールのみ/物理強制） | 効くか（再発実績に照らして） |
|----|----------------------------------|------------------------------|
| … | 物理強制（PreToolUse hook で迂回検出） | ◎（経路上・YES/NO・低誤検知） |
| … | 文書化のみ | ✗（5/5再発の型・単独採用しない） |

## 採用する決定論ガードレール（最低1つ）
- 機構: [hook / 回帰テスト / 正準経路強制 / lint のどれか]
- 設置位置: [作業がそこを通らないと完了できない位置]
- 判定: [決定論YES/NO。証跡有無・1円一致・0エラー等]
- fail-open・低誤検知の担保: [重ゲート自壊（2026-05-26）を踏まないこと]
```

却下リスト（採用してはいけない）: 「気をつける」「次回から確認する」「ルールに追記するだけ」「ツールを作るだけで使用を強制しない」。

→ **Gate-3（AskUserQuestion）**: 「この決定論ガードレールを実装し、bd issue として登録してよいですか？」
（OK → Step 7 へ。NG → 対策を再設計）

## Step 7: ループクローズと記録（ここまでやって初めて完了）

### 7-1. bd issue 登録（必須）

当該プロジェクトの bd（beads）に、ガードレールを追跡 issue として登録する。

```bash

command -v bd >/dev/null && test -d .beads && echo "bd OK" || echo "bd 未初期化"

# 登録（例）。即IDだけ欲しいときは `bd q "<title>"`
bd create "<ガードレール名>" \
  -t task -p 2 \
  -d "<症状＋真因の要約>" \
  --acceptance "<ガードレールが配線・テストされ、本インシデントを再現すると検出/ブロックする状態>" \
  --labels ai-misconduct,guardrail,<A|B|C>
```

- 出力された issue ID を控える。**incident はこの issue が close されるまで open 扱い**（実装まで追跡）。

- **bd 未初期化フォールバック**（当該PJに `.beads/` が無い＝sd003 本体等）:
  - `bd init` をユーザーに提案（Gate-3 の承認範囲で）、または

## 参考: 模範5Why（at002 2026-06-13 — このレベルと構造を目指す）

> **問題**: 最初から決定論（既存スキル `journal-builder` + `yayoi-csv-format`）で弥生CSVを書かず、自分で `csv.writer` を手書きした。結果、引用の有無・髙のバイト形・経過措置など現場固有の落とし穴を全部踏み、**ユーザーに4回もエラーログを取らせた**。

| | Why | 答え |
|---|-----|------|
| Why1 | なぜ決定論（既存スキル）を使わなかったか | 弥生CSV書き出しの共通スキル `skills/common/yayoi-csv-format.js` の存在を着手時に確認せず、`csv.writer` で自作した |
| Why2 | なぜ確認しなかったか | 「25列に値を並べるだけの単純作業」とみなし `skills/` を探索しなかった。本当の難所（弥生が受理する書式の固有知）を見落とした |
| Why3 | なぜ単純作業とみなしたか | 弥生フォーマットを「知っている」という確信。validate が0エラーで通ったことで**「動いた」と誤認**し、実機取込前に完成扱い（Work First違反：検証通過≠実データ完走） |
| Why4 | なぜ一般知識を一次情報にし、原典を下に置いたか | 「引用が要る／髙はNEC形だと弾かれる／経過措置は二重控除」は一般的な弥生知識に無い、**事故からしか得られない固有知**。その層が自分に欠けていることに気づけなかった（unknown unknown）。`taka.txt` という最高の原典をもらってもなお自作理論を上に置いた |
| **Why5（真因）** | なぜ過信が起きたか | **優先順位の逆転**。私の中で「自分の一般知識 ＞ スキル ＞ 実物の証拠」になっていた。正しくは「**実物の証拠 ＞ スキル（固有知の結晶）＞ 自分の知識**」 |

**真因の正体**: 会計の決定論（1円も狂わせない）は機械＝スキルの領域なのに、非決定論＝自分の推測・理論でやろうとした。[[at002-llm-self-driving]]（決定論への収束）と真逆で、CLAUDE.md 冒頭「スキル至上主義＝自分の知識でスキルを代替するな（全違反の真因＝過信）」そのもの。「弥生公式」の捏造も同根＝原典を確認せず手元の紙を権威化した。**原典照合の欠如という単一の真因**。

**対策（「気をつける」は対策にあらず＝機構で固める）**:
1. 弥生CSVの書き出しは `yayoi-csv-format.js` 経由を唯一の正準とし、手書き `csv.writer` を禁止 → 評価器が「自作書き出しでないか」を判定
2. `ref-弥生仕訳ルール` に「書き出し器の正準＝`yayoi-csv-format.js`」を明記し、弥生CSV生成前に force-read
3. 「公式」と言う前に原典（URL／実機エクスポート）を必ず提示。出せないものを「公式」と呼ばない
4. **実物 ＞ スキル ＞ 自分の知識** を固定し、実物と自分の理論が食い違ったら無条件で実物を採る
