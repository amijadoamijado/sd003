# Real Data First（実データ先行・テスト最小主義）

## 原則

> 実データまたはコピーで直接動かす。モック経由しない。
> テストは「本番で発生したバグを再現・固定する」ためだけに書く。
> **テストのためのテストは絶対禁止。やるだけ無駄。**

## 背景

テストを先に書く → モック前提の設計になる → 本番データの「汚さ」を想定できない → 本番で動かない。
これを繰り返してきた。「カバレッジ80%」目標がテストのためのテストを量産する原因だった。

ユーザー明言:
> 「テストのためのテストは絶対にするな。やるだけ無駄だ。実データで修正したほうがよっぽど効率がよい」

## 適用

### 動作確認の順序

```
1. 本番データ（またはそのコピー）を用意する
2. 実データで直接動かす（モックを経由しない）
3. 実データで動いている状態を確認する
4. 動かない箇所・バグが出たら修正する
5. 本番で再発防止が必要なバグだけ、最小テストを追加する
```

### テストを書くタイミング

| 状況 | テストを書くか |
|------|--------------|
| 実装完了後のカバレッジ担保 | ❌ 書かない（テストのためのテスト） |
| 新規機能の動作確認 | ❌ 実データで動作確認（テストではない） |
| 実データで発生したバグを再現 | ✅ 書く（バグを固定する最小テスト） |
| リグレッション防止 | ✅ 書く（ただし実データで再現可能なもののみ） |
| 外部 API の応答パターン検証 | ✅ 書く（本番データコピーで） |

### 禁止パターン（VTD-001〜005 自動検出）

以下のテストは pre-commit hook で自動拒否される:

| VTD | 禁止パターン | 例 |
|-----|------------|-----|
| VTD-001 | 空配列・空オブジェクトでテスト | `headers: []` |
| VTD-002 | 全値が空/ゼロのオブジェクト | `{ name: '', amount: 0 }` |
| VTD-003 | アサーションなしテスト | `toBeDefined()` のみ |
| VTD-004 | 自明な真偽テスト | `expect(true).toBe(true)` |
| VTD-005 | フォールバック付きテスト | `try { ... } catch { /* skip */ }` |

### カバレッジ目標の廃止

`.claude/rules/global/quality-standards.md` のカバレッジ80%目標は**廃止**する。
代わりに以下を完了条件とする:

| 旧基準 | 新基準 |
|--------|--------|
| カバレッジ80% | 実データで動作することをブラウザで確認 |
| 全テスト通過 | VTD-001〜005 通過 + 実データ動作確認 |
| 単体/統合/E2E 各層 | 本番バグに対する最小テスト + Antigravity E2E（該当時） |

### Adapter 層のテスト

Adapter 層のみは本番データコピーでの徹底テストが必要（`.claude/rules/testing/production-data-tdd.md`）。
これは「テストのためのテスト」ではなく「本番データの汚さを検証する」ため。

本番データ例:
- 結合セル、空白行・列
- 全角半角混在、日付形式バラバラ
- 予期しないヘッダー、数値に見える文字列

これらの検証は必須。モックでは再現不可能だから。

## 禁止事項

| 禁止 | 理由 |
|------|------|
| モック・ダミー・空データでのテスト | 本番を再現しない |
| カバレッジ数値のためのテスト | テストのためのテスト |
| `toBeDefined()` だけのアサーション | 値を検証していない |
| `expect(true).toBe(true)` 等の自明テスト | 何も検証していない |
| フォールバック付きテスト | エラーを隠蔽 |
| 実装前にテストを書く（TDD 正攻法） | Adapter 以外では機能しない |
| E2E で 本番URLが見えなくなるテスト | 実データが届かない |

## 例外: Adapter 層

Adapter 層だけは本番データコピーでの徹底検証が必要。
これは「変換層の健全性」という明示的目的があるため、柱3違反ではない。

詳細: `.claude/rules/testing/production-data-tdd.md`
