# Work First - まず動かす原則

## 最上位原則（全ルールに優先）

> コードは動いて初めて価値がある。
> 型定義79個、テスト442個、レビュー3ラウンド — 動かなければ全て無価値。

## 開発順序（厳守）

```
1. 最小コードで動かす（50行でいい）
2. 実環境で動作確認する（ブラウザで開いて操作する）
3. 動いたら、その動作を保証するテストを書く
4. 必要になったら抽象化する
5. 安定したらプロセス文書を書く
```

**禁止される順序:**
```
❌ 型定義 → インターフェース → アダプター → コア → テスト → レビュー → 動かない
```

## 動作確認の義務（省略不可）

### GAS Web App の場合
```
コード変更 → clasp push → ブラウザで開く → 操作する → 動くか確認
```

- `clasp push` 後、必ずブラウザで実画面を開いて操作する
- claude-in-chrome または手動で確認する
- 「動くはず」は確認ではない。「動いた」が確認

### 確認せずに次に進むことの禁止

| やってはいけないこと | なぜ |
|---------------------|------|
| テスト全パスで安心して実画面を開かない | モックの中でモックが正しいだけ |
| Codexレビュー通過で安心して実画面を開かない | コード読みだけでは動作は保証されない |
| 型チェック通過で安心して実画面を開かない | 型が正しくても実行時に動かない |

## 変更前の3点固定（必須）

作業を始める前に以下を明確にする:

| # | 項目 | 例 |
|---|------|-----|
| 1 | 運用ルール | clasp push のみ、deploy禁止 |
| 2 | 反映方法 | push → @HEAD で確認 |
| 3 | 確認対象URL | `https://script.google.com/macros/s/XXX/exec` |

## 変更仮説プロトコル（必須）

コード変更前に以下を明文化する:

```
症状: [何が起きているか]
仮説: [なぜそうなるか]
確認方法: [どう検証するか]
失敗時の次手: [仮説が外れたら何をするか]
```

**原因確定前に修正を反映して回すことの禁止。**

## 抽象化の判断基準

| 状態 | やるべきこと |
|------|-------------|
| まだ動いていない | ベタ書きで動かす |
| 1つの場所で動いている | そのまま。抽象化しない |
| 2箇所で同じコードが出た | まだ様子見 |
| 3箇所で同じコードが出た | 抽象化を検討する |

**最初から Adapter-Core 分離、Env Interface、Railway型を導入しない。**
動くものができてから、必要に応じて導入する。

## プロセス文書の判断基準

| ソフトウェアの状態 | プロセス文書 |
|-------------------|------------|
| まだ動いていない | 書かない |
| 動いているが不安定 | 最小限（DONE.mdのみ） |
| 安定して動いている | 必要に応じて追加 |

**動かないソフトウェアの管理文書は無価値。**

## 変更粒度ルール

- 1修正ごとに動作確認する
- 複数の挙動を一度に変えない
- 何が効いたか分からない変更は禁止

## GAS固有の注意事項

知っている制約は即座に実装に反映する:

| 既知の制約 | 実装への反映 |
|-----------|-------------|
| iframe内で `window.location.href` が動かない | `google.script.run` + `google.script.host.close()` を使う |
| `script.googleusercontent.com` でCORSが異なる | fetch先を適切に設定する |
| @HEAD と固定 deployment の挙動差 | 確認対象URLを事前に固定する |

**「知っている」と「実装に反映する」は別。反映して初めて価値がある。**

## このルールの位置づけ

- **全ルールに優先する**
- テスト基準、品質基準、AI協調ワークフロー — 全て「動くもの」が存在することが前提
- 動くものがない段階でこれらのルールを適用しても意味がない
