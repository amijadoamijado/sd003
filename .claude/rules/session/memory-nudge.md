# メモリナッジ（Periodic Memory Evaluation）

## 原則

> 大きなタスク完了後、「この知見を保存すべきか」を自己評価する。
> クラッシュや新セッション開始で失われる前に、価値ある知見を永続化する。

## 背景

Hermes Agent（Nous Research）の「periodic nudge」パターンを参考にSD003へ導入。
Hermesではエージェントが定期的に「何を記憶すべきか」を自律評価する。
SD003ではClaude Codeのルールとして、タスク完了時に自己評価を行う。

## トリガー条件

以下のいずれかが完了した直後に自己評価を実行する:

| トリガー | 例 |
|---------|-----|
| テスト全通過 | `npm test` → all pass |
| 実装完了 | 機能実装が一区切りついた時 |
| デバッグ解決 | エラーの根本原因を特定・修正した時 |
| ユーザー修正2回以上 | 同じ種類の修正を繰り返し受けた時 |

## 自己評価の観点

以下を順にチェックする（所要時間: 数秒）:

1. **ユーザー修正**: セッション中にユーザーから修正指示を受けたか？
   - 受けた → その修正パターンはルール化すべきか？
2. **予想外の解決方法**: 通常とは異なるアプローチで解決したか？
   - はい → その方法は再利用可能か？
3. **新発見のパターン**: プロジェクト固有の制約やパターンを発見したか？
   - はい → 他のセッションでも有用か？
4. **外部サービスの挙動**: API・ツールの未文書化の挙動を発見したか？
   - はい → 記録すべきか？

## 保存先の判断基準

| 条件 | 保存先 | 方法 |
|------|--------|------|
| 今後の全プロジェクトで使える | auto-memory | `~/.claude/projects/*/memory/` にWrite |
| SD003プロジェクトのみで有用 | sessionwrite備考 | `/sessionwrite` の備考セクションに記載 |
| 一時的な作業メモ | 保存不要 | 記録しない |

## learning-nudgeとの責務分担

| ルール | 担当 | タイミング | 動作 |
|--------|------|-----------|------|
| **memory-nudge**（本ルール） | 知見の直接保存 | タスク完了後 | auto-memoryに直接書き込み可 |
| **learning-nudge** | 修正パターンの提案 | sessionwrite時 | 提案のみ（書き込み禁止） |

memory-nudgeは「自己判断で保存」、learning-nudgeは「ユーザーに提案」という違い。

## 実行ルール

- **非対話**: AskUserQuestionは使わない。自己完結で判断する
- **非ブロッキング**: 評価に30秒以上かけない
- **控えめ**: 「全部保存」ではなく「本当に価値があるものだけ」保存
- **重複チェック**: auto-memoryに書く前に既存エントリとの重複を確認

## 禁止事項

| 禁止 | 理由 |
|------|------|
| 毎タスク完了で必ず保存 | ノイズが増えて価値が薄まる |
| コード内容をそのまま記録 | コードはgitで管理される |
| 一般的な知識を記録 | LLMの知識で十分なものは不要 |
| ユーザーに保存の可否を毎回聞く | 非対話原則に違反 |
