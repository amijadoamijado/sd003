# Output Primacy（アウトプット最優先）

## 原則

> 「完了」の定義は、ユーザーが見る画面・受け取る成果物が**存在し検証済みの状態**である。
> ファイル数、テスト数、型定義数、コミット数は完了の指標ではない。

## 背景

cr001 案件で Gemini が97ファイル生成し「実装完了」扱いになった。画面は一度も存在しなかった。
内部成果物（ファイル、型、テスト）を積み上げても、ユーザーが触れるものがゼロならゼロ。
この失敗を繰り返さないため、「完了」の定義をアウトプット存在に固定する。

## 適用

### 完了判定

| 状態 | 判定 |
|------|------|
| 画面/成果物が存在し、ユーザーが確認した | **完了** |
| 画面/成果物が存在するが、ユーザー未確認 | 未完了（User Confirmation 待ち） |
| ファイル多数・テスト全通過・画面なし | **未着手** |
| 型定義・interface 完備・画面なし | **未着手** |

内部成果物は「未着手に対する過剰投資」。画面化が先。

### IMPLEMENT_REQUEST の必須項目

IMPLEMENT_REQUEST.md の Section 2 は必ず以下で始まる:

```markdown
## 2. ゴール（ユーザーが見る画面・受け取るもの）

{ここに "完成時にユーザーが見るもの" を書く。ファイル一覧ではない}
```

この欄が空の IMPLEMENT_REQUEST は**作成できない**（Template Reject により拒否）。

### 進捗レポート

進捗報告には以下のいずれかが必要:
- 画面のスクリーンショット（`materials/images/{案件ID}/`）
- 成果物ファイル（`materials/` 配下）
- 実データでの動作ログ

これらを含まない進捗報告は「未着手」と同等に扱う。

### コードレビュー配点

Codex レビューの REVIEW_REPORT.md での配点:

| 項目 | 配点 |
|------|------|
| UI/アウトプット品質（視覚的評価 70点満点） | **60%** |
| 機能動作（実データで動くか） | **30%** |
| 内部コード品質（型、命名、構造） | **10%** |

「内部コード品質が完璧だが画面が成立していない」は **Request Changes** 確定。

## 禁止事項

| 禁止 | 理由 |
|------|------|
| 画面がない状態で「実装完了」と報告する | 定義違反 |
| 内部ファイル数をもって進捗と主張する | 定義違反 |
| スクリーンショットを含まない進捗報告 | 検証不能 |
| IMPLEMENT_REQUEST の「画面」欄を空にする | 入口ガードレール違反 |
| 型定義・interface の完成度を完了指標に使う | 柱2の領域（内部は黙れ） |

## 例外

以下はアウトプット＝画面ではないため、画面必須の対象外:
- CLIツール（アウトプットは stdout + 動作ログ）
- ライブラリ（アウトプットは API仕様書 + 使用例）
- データ変換スクリプト（アウトプットは変換結果ファイル）

ただし**成果物として検証可能な形**は必須（stdout ログ、README、出力ファイル等）。

## 全AIモデル共通

このルールはClaude Code、Codex、Gemini CLI、Antigravity全てに適用される。
