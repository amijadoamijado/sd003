# Find Duplicates - セマンティック重複検出

## 概要

コピー＆ペースト検出ツールでは見つけられない「同じ意図の異なる実装」を発見する。
構文的には異なるが意味的に同じ処理を行うコードを特定し、統合候補として報告する。

## 2フェーズアプローチ

### Phase 1: 古典的関数抽出

コードベースから関数・メソッドを機械的に抽出する。

```
1. 対象ディレクトリの全ソースファイルを列挙
2. 各ファイルから関数/メソッド定義を抽出
3. シグネチャ・行数・パラメータ数を記録
4. 類似シグネチャのペアを候補として列挙
```

#### 抽出対象

| 言語 | パターン |
|------|---------|
| TypeScript | `function`, `=>`, `class method` |
| JavaScript | `function`, `=>`, `prototype` |

#### 類似性の指標

| 指標 | 閾値 |
|------|------|
| パラメータ数が同じ | 候補 |
| 戻り値の型が同じ | 候補 |
| 行数が±20%以内 | 候補 |
| 関数名が類似（レーベンシュタイン距離） | 候補 |

### Phase 2: LLM駆動の意図クラスタリング

Phase 1の候補ペアをLLM（Claude）で意図分析する。

```
1. 候補ペアのソースコードを読み込む
2. 各関数の「意図」を自然言語で要約
3. 意図が同じペアをクラスタリング
4. 統合の推奨度を判定
```

#### 判定カテゴリ

| カテゴリ | 説明 | アクション |
|---------|------|-----------|
| **Identical Intent** | 同じ意図、同じ入出力 | 統合を強く推奨 |
| **Similar Intent** | 似た意図、微妙な差異 | 統合を検討 |
| **Related Intent** | 関連するが異なる意図 | 共通基盤の抽出を検討 |
| **Different** | 異なる意図 | 統合不要 |

## 実行方法

### 基本実行

```
対象ディレクトリを指定して実行:

1. Agent(Explore) で対象ディレクトリの関数一覧を抽出
2. 類似候補ペアを特定
3. 各ペアの意図を分析
4. レポートを出力
```

### `/refactor:init` からの連携

`/refactor:init` の Pattern Agent が本スキルを参照して重複検出を実行:

```
/refactor:init {scope}
  └── Pattern Agent
        └── find-duplicates スキルを適用
              ├── Phase 1: 関数抽出
              └── Phase 2: 意図クラスタリング
```

## 出力フォーマット

```markdown
## 重複検出レポート

### Identical Intent（統合推奨）
| ファイルA | 関数A | ファイルB | 関数B | 意図 |
|----------|-------|----------|-------|------|
| src/a.ts | formatDate() | src/b.ts | toDateString() | 日付→文字列変換 |

### Similar Intent（統合検討）
| ファイルA | 関数A | ファイルB | 関数B | 差異 |
|----------|-------|----------|-------|------|
| src/a.ts | validate() | src/c.ts | check() | エラーメッセージのみ異なる |

### 統計
- 対象関数数: N
- 検出ペア数: M
- Identical: X
- Similar: Y
```

## 制限事項

| 制限 | 理由 |
|------|------|
| 100関数以上は分割実行推奨 | コンテキストウィンドウの制約 |
| 外部ライブラリは対象外 | node_modules内は検査しない |
| テストファイルは別扱い | テストの重複はプロダクションと異なる判断基準 |

## Work First原則との関係

重複検出は「動くソフトウェア」が存在する段階で実行する。
動かないコードの重複を検出しても統合の判断ができない。

```
❌ コード未完成 → 重複検出 → 統合
✅ コード動作確認済み → 重複検出 → 統合検討 → 動作再確認
```
