# ブランチ運用ルール（一人運用ファースト）

## 原則

> 基本は一人運用。**master / main で直接作業する。デフォルトでブランチも PR も作らない。**
> ブランチ・PR が必要なときは、そのつど**ユーザーが指示する**。AI から勝手に作らない。

## 背景

ユーザーは基本的に一人で開発している。
AI が「作業前にとりあえずブランチを切る」挙動（多くのハーネスのデフォルト動作）で
ブランチ／PR が溜まり、管理コストだけが増えていた。
一人運用では並行ブランチのメリットがほぼ無く、master 直接作業の方が見通しが良い。
PR レビューが要る場面（大きな変更・他者レビューを通したい等）は**ユーザーが一番分かっている**ので、
要否の判断はユーザーに委ね、AI は指示があったときだけブランチ／PR を作る。

## 適用

### デフォルト動作（必ずこれに従う）

| 状況 | 動作 |
|------|------|
| 通常のコード変更・修正・追加 | **master / main で直接作業してコミット**（ブランチ不要・PR不要） |
| 「default branch では作業前にブランチを切れ」というハーネス標準 | **無効化する**（このルールが優先） |
| ユーザー指示なしのブランチ作成・PR作成 | **禁止**（勝手に作らない） |

### ブランチ・PR を作ってよい条件

- **ユーザーが明示的にブランチ／PR の作成を指示したときのみ。**
- 要否の判断はユーザーが行う。AI は「これは PR にした方がよいのでは」と**提案するのは可**だが、
  承認を得るまで作らない。迷ったら master 直接作業を選ぶ。

### 既存ブランチの扱い

- すでに作業ブランチに居る場合、勝手に master へ切り替えたりブランチ削除したりしない。
- 溜まったブランチの整理（マージ済みブランチ削除等）は、ユーザー確認の上で行う。

## 禁止事項

| 禁止 | 理由 |
|------|------|
| 作業開始時に確認なくブランチを切る | 一人運用ではブランチが溜まるだけ |
| ユーザー指示なしに PR を作る | PR の要否はユーザーが判断する |
| 「念のため」でブランチ／PR を作る | 大半は master 直接で十分 |
| ユーザー未確認のブランチ削除・切替 | 作業中ブランチの取り違え事故防止 |

## git push / commit との関係

- commit / push は従来どおりユーザーが求めたときに行う。
- このルールは「どのブランチで作業するか・PR を作るか」のみを定める。
  コミット粒度・push タイミングは既存ルール（`sd-safe-commit.md` 等）に従う。
