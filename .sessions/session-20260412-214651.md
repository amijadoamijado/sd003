### 備考

**本セッションの構造的ポイント**:

ユーザーとの7回のやりとりから、SD003 の根本問題が**ルール不足ではなく構造不足**であることが確定した。
既存ルールは「書いてある」だけで実行パイプラインに浸透していなかった。cr001 案件で97ファイル生成・画面ゼロで「完了」扱いになった構造的失敗の根本原因。

本セッションでは「ルールではなく **ガードレール** を作る」という方針で 4本柱ドクトリンを制定し、T1-T5 5種のガードレール分類を導入した:

- **T1 Template Reject**: 必須欄が空なら拒否（今回 Phase B で実装）
- **T2 Pre-commit Block**: 違反コードをコミット前に拒否（次セッション Phase D）
- **T3 Workflow Stop-Hook**: 条件未達で次フェーズ進行不可（次セッション Phase D）
- **T4 Command Guard**: コマンド自体が条件強制（今回 Phase C で実装）
- **T5 Observability**: 違反を可視化・学習（今回 Phase C で実装）

次セッションで Phase D を実装すれば、`any` 型や `@ts-nocheck`、スクショなしでの review 進行が**物理的に不可能**になる。

**.sd/ gitignore 問題の再発**:

途中で `.sd/ai-coordination/workflow/templates/` に templates を作成したが、post-commit hook が `.sd/` を HEAD から auto-restore する挙動でテンプレートが wipe された。
対応として templates の正本を `.claude/templates/workflow/`（tracked パス）に移した。`.sd/ai-coordination/` は per-project スクラッチ領域として維持。
