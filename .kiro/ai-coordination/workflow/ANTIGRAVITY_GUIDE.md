# Antigravity テスト運用ガイド

## 概要

AntigravityはSD003プロジェクトにおいてE2Eテスト・探索的テスト・本番確認を担当します。
Codexのレビュー Approve 後、自動連鎖で `/workflow:test` が実行され、TEST_REQUESTが作成されます。

### コマンド運用

Antigravityはブラウザベースのテスト実行環境です。
テスト依頼は `.kiro/ai-coordination/workflow/spec/{案件ID}/` に配置されます。

- テスト依頼読み込み: `TEST_REQUEST_{NNN}.md` を参照
- テスト報告作成: `TEST_REPORT_{NNN}.md` を作成

---

## 1. 対応するテスト種別

| 種別 | 入力 | 出力 |
|------|------|------|
| E2Eテスト | TEST_REQUEST_{NNN}.md | TEST_REPORT_{NNN}.md |
| 探索的テスト | テスト依頼 or 口頭指示 | TEST_REPORT_{NNN}.md |
| 本番確認 | デプロイ後確認依頼 | TEST_REPORT_{NNN}.md |

## 2. E2Eテスト

### 2.1 入力ファイル
```
.kiro/ai-coordination/workflow/spec/{案件ID}/TEST_REQUEST_{タスク番号}.md
```

### 2.2 テスト実行フロー

```
1. TEST_REQUEST_{NNN}.md を読み込む
2. 対象環境にアクセス（URL / 認証情報を確認）
3. テストケースを順次実行
4. 各テストケースの結果を記録（Pass / Fail）
5. スクリーンショットを取得（証跡ID付き）
6. TEST_REPORT_{NNN}.md を作成
7. handoff-log.json に記録
```

### 2.3 テスト観点（7項目）

#### 2.3.1 機能正常動作
- 実装指示の要件を満たしているか
- 期待通りの入出力が行われるか
- エラーケースが適切に処理されるか

#### 2.3.2 UI表示・レイアウト
- 画面レイアウトが崩れていないか
- レスポンシブ対応は適切か
- 文字化けや表示欠けがないか

#### 2.3.3 データ整合性
- 入力データが正しく保存されるか
- 画面遷移後もデータが維持されるか
- 既存データへの影響がないか

#### 2.3.4 パフォーマンス
- ページ読み込みが許容範囲内か
- 大量データ処理で遅延がないか
- メモリリークの兆候がないか

#### 2.3.5 エラーハンドリング
- エラーメッセージが適切に表示されるか
- ネットワークエラー時の動作
- 不正入力時の動作

#### 2.3.6 アクセシビリティ
- キーボード操作が可能か
- フォーカス管理が適切か
- 色コントラストが十分か

#### 2.3.7 セキュリティ（基本）
- 認証バイパスの可能性がないか
- XSSの兆候がないか
- 機密情報の露出がないか

## 3. 探索的テスト

### 3.1 概要
仕様外の動作を確認するテスト。TEST_REQUESTに「探索的テスト」セクションがある場合に実施。

### 3.2 実施手順
```
1. 対象機能の正常フローを確認
2. 以下の観点で「壊す」テスト:
   - 想定外の入力（空文字、超長文、特殊文字）
   - 想定外の操作順序（戻る→進む→リロード）
   - 並行操作（別タブで同時操作）
3. 発見した問題を記録
```

### 3.3 記録対象
- 予期しないエラーメッセージ
- UI崩れ
- データ不整合
- UX上の問題点

## 4. 報告書作成

### 4.1 出力ファイル
```
.kiro/ai-coordination/workflow/review/{案件ID}/TEST_REPORT_{タスク番号}.md
```

### 4.2 テンプレート
`.kiro/ai-coordination/workflow/templates/TEST_REPORT.md` を使用

### 4.3 判定基準

| 判定 | 条件 |
|------|------|
| Pass | 全必須テストケース通過、クリティカルな問題なし |
| Fail | 必須テストケースに1つ以上の失敗あり |
| Blocked | テスト実行不可能（環境問題、認証エラー等） |

### 4.4 判定の詳細

#### Pass の条件
- 全必須テストケース（P0/P1）が通過
- クリティカルなUI崩れがない
- データ整合性が保たれている
- P2テストケースの失敗は許容（報告は必須）

#### Fail の条件
- P0テストケースが1つでも失敗
- P1テストケースが2つ以上失敗
- データ破壊やセキュリティ問題を発見
- 機能の主要フローが動作しない

#### Blocked の条件
- テスト対象環境にアクセスできない
- 認証エラーでログインできない
- 前提条件（テストデータ等）が整っていない
- デプロイが完了していない

## 5. 完了後の連絡

### 5.1 Pass時
```markdown
## テスト完了: Pass

- **案件ID**: {案件ID}
- **タスク番号**: {タスク番号}
- **テスト結果**: Pass
- **成功率**: X/Y テストケース通過
- **レポート**: .kiro/ai-coordination/workflow/review/{案件ID}/TEST_REPORT_{タスク番号}.md
- **次のアクション**: Phase 8（工程完了）へ
```

### 5.2 Fail時
```markdown
## テスト完了: Fail

- **案件ID**: {案件ID}
- **タスク番号**: {タスク番号}
- **テスト結果**: Fail
- **失敗テスト**: X件（P0: A件, P1: B件）
- **レポート**: .kiro/ai-coordination/workflow/review/{案件ID}/TEST_REPORT_{タスク番号}.md
- **次のアクション**: 修正対応（Gemini CLI → Codexレビュー → 再テスト）
```

### 5.3 Blocked時
```markdown
## テスト完了: Blocked

- **案件ID**: {案件ID}
- **タスク番号**: {タスク番号}
- **テスト結果**: Blocked
- **ブロック理由**: {具体的な理由}
- **レポート**: .kiro/ai-coordination/workflow/review/{案件ID}/TEST_REPORT_{タスク番号}.md
- **次のアクション**: ブロック解消後に再テスト
```

## 6. チェックリスト

### 6.1 テスト開始前
- [ ] TEST_REQUEST_{NNN}.md を全文読んだ
- [ ] 対象環境にアクセスできることを確認した
- [ ] 必要な認証情報を確認した
- [ ] テストデータが用意されていることを確認した
- [ ] 実装指示（IMPLEMENT_REQUEST）を確認した

### 6.2 テスト実行中
- [ ] 全必須テストケースを実行した
- [ ] 各テストケースの結果を記録した
- [ ] 必須スクリーンショットを取得した
- [ ] 発見した問題を記録した
- [ ] 探索的テスト（指定がある場合）を実施した

### 6.3 テスト報告作成時
- [ ] テンプレートに沿って報告書を作成した
- [ ] 全テストケースの結果（Pass/Fail）を記載した
- [ ] スクリーンショットを添付した（証跡ID付き）
- [ ] 判定（Pass/Fail/Blocked）を明記した
- [ ] 推奨事項を記載した
- [ ] handoff-log.json に記録した

## 7. トラブルシューティング

### Q: 対象環境にアクセスできない
1. URLが正しいか確認
2. VPN/ネットワーク設定を確認
3. Blocked として報告

### Q: テストデータが存在しない
1. TEST_REQUEST の前提条件を確認
2. Claude Code に確認依頼
3. 前提条件未整備として Blocked 報告

### Q: テスト中にエラーが発生した
1. エラーメッセージをスクリーンショット
2. コンソールログを確認
3. Fail として報告し、エラー内容を詳記

### Q: 判断に迷う場合
1. TEST_REQUEST の成功条件を再確認
2. 不明点は Claude Code に確認依頼
3. 保守的な判定（Fail）を選択

- 保存先は evidence/ ディレクトリ
