### 2.2 Ultrathink統合分析

```
入力: 3エージェントの調査結果

処理:
1. エージェント間の相互参照
   - Spec Agent: 期待動作 ←→ Code Agent: 実際の動作
   - Code Agent: 問題箇所 ←→ Solution Agent: 修正案

2. 矛盾・ギャップの特定
   - 仕様と実装の乖離
   - 複数仮説の整合性

3. 仮説の優先順位付け
   - 証拠の強さ
   - 修正の容易さ
   - リスクレベル

出力:
- 統合分析レポート
- 日本語解説（必須）
- 乖離分類（v2.4）
```

### 2.2.1 乖離分類ロジック（v2.4新機能）

```
┌────────────────────────────────────────────────────────────────┐
│                 Divergence Classification Logic                 │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│  入力: Ultrathink統合分析結果                                   │
│                                                                 │
│  判定基準:                                                      │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ 1. 仕様書の最終更新日 vs コードの最終変更日             │    │
│  │    - Spec > Code → Code Defect 傾向                     │    │
│  │    - Spec < Code → Spec Defect 傾向                     │    │
│  ├─────────────────────────────────────────────────────────┤    │
│  │ 2. コードの動作が「意図的改善」か「バグ」か             │    │
│  │    - 改善 → Spec Defect                                 │    │
│  │    - バグ → Code Defect                                 │    │
│  ├─────────────────────────────────────────────────────────┤    │
│  │ 3. 他コードとの整合性                                   │    │
│  │    - 高い → コードの動作が正しい可能性                  │    │
│  │    - 低い → コードが逸脱している可能性                  │    │
│  ├─────────────────────────────────────────────────────────┤    │
│  │ 4. ビジネスロジックの妥当性                             │    │
│  │    - コードが妥当 → Spec Defect                         │    │
│  │    - 仕様が妥当 → Code Defect                           │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  分類結果:                                                      │
│  ┌──────────────┬──────────────────────────────────────────┐    │
│  │ Code Defect  │ 仕様が最新かつ明確、コードが逸脱         │    │
│  │              │ → Step 3へ（従来フロー）                 │    │
│  ├──────────────┼──────────────────────────────────────────┤    │
│  │ Spec Defect  │ コードの動作が合理的、仕様が古い/曖昧    │    │
│  │              │ → Step 2.6へ（仕様書更新提案）           │    │
│  ├──────────────┼──────────────────────────────────────────┤    │
│  │ Both Defect  │ 両方に問題あり                           │    │
│  │              │ → 優先順位付けて両方対応                 │    │
│  ├──────────────┼──────────────────────────────────────────┤    │
│  │ Ambiguous    │ 判断困難                                 │    │
│  │              │ → ユーザー確認                           │    │
│  └──────────────┴──────────────────────────────────────────┘    │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

### 2.2.2 仕様書更新提案フロー（v2.4新機能）

```
Spec Defect 判定時のフロー:

Step 2.5 (Divergence Classification: Spec Defect)
    │
    ▼
Step 2.6: Spec Update Proposal
    │
    ├── 更新対象ファイル・セクション特定
    ├── 現在の記述 vs 提案する記述
    └── 影響範囲分析
    │
    ▼
[AskUserQuestion: Spec更新を承認？]
    │
    ├─ Yes → 仕様書更新実行 → 完了
    │
    └─ No (Override: Code Defect)
         │
         ▼
    Step 3へ（従来のCode Fixフロー）
```

### 2.4 バグレポート管理

#### Bug ID フォーマット
```
BUG-{PROJECT}-{SEQ}-{TIMESTAMP}

例: BUG-SD002-001-202601031530

- PROJECT: プロジェクトID（大文字）
- SEQ: 連番（3桁）
- TIMESTAMP: yyyymmddhhmm
```

#### ファイル構造
```
docs/troubleshooting/
├── bug-reports/
│   ├── BUG_INDEX.md          # インデックス
│   ├── BUG-SD002-001-*.md    # 個別レポート
│   └── BUG-SD002-002-*.md
└── RESOLUTION_LOG.md         # 解決ログ
```

## 3. インターフェース設計

### 3.1 入力インターフェース

| Step | 入力方法 | 形式 |
|------|---------|------|
| Step 1 | AskUserQuestion | エラーログ/症状説明 |
| Step 2-7 | AskUserQuestion | 選択肢（Yes/No/Modify） |

### 3.2 出力インターフェース

| 出力 | 形式 | 必須要素 |
|------|------|---------|
| エージェント報告 | Markdown表 | 発見内容、信頼度 |
| 統合分析 | Markdown + 日本語 | 統合分析解説 |
| ASCII図 | ASCII art + 日本語 | 各図の解説 |
| 仮説 | Markdown表 | 順位、証拠、信頼度 |
| バグレポート | Markdown | 全セクション必須 |
