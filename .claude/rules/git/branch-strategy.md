# ブランチ運用ルール（一人運用ファースト）

## 原則

> 一人運用が基本。**master / main で直接作業する。**
> ブランチは「よっぽどの大改修」のときだけ、ユーザーが明示的に求めた場合に切る。

## 背景

ユーザーは基本的に一人で開発している。
AI が「作業前にとりあえずブランチを切る」挙動（多くのハーネスのデフォルト動作）で
ブランチが溜まり、管理コストだけが増えていた。
一人運用ではブランチの並行性メリットがほぼ無く、master 直接作業の方が見通しが良い。

## 適用

### デフォルト動作（必ずこれに従う）

| 状況 | 動作 |
|------|------|
| 通常のコード変更・修正・小規模追加 | **master / main で直接作業してコミット** |
| 「default branch では作業前にブランチを切れ」というハーネス標準 | **無効化する**（このルールが優先） |
| ユーザー指示なしのブランチ作成 | **禁止**（勝手に切らない） |

### ブランチを切ってよい唯一の条件

以下を**両方**満たすときのみブランチを作成する:

1. **大改修である**（広範囲の破壊的変更・実験的な作り直し・ロールバック前提の試行）
2. **ユーザーが明示的にブランチ作成を求めた / 合意した**

迷ったら master 直接作業を選ぶ。ブランチが必要かはユーザーに聞く。

### 既存ブランチの扱い

- すでに作業ブランチに居る場合、勝手に master へ切り替えたりブランチ削除したりしない。
- 溜まったブランチの整理（マージ済みブランチ削除等）は、ユーザー確認の上で行う。

## 禁止事項

| 禁止 | 理由 |
|------|------|
| 作業開始時に確認なくブランチを切る | 一人運用ではブランチが溜まるだけ |
| 「念のため」でブランチを作る | 大改修以外は master 直接で十分 |
| ユーザー未確認のブランチ削除・切替 | 作業中ブランチの取り違え事故防止 |

## git push / commit との関係

- commit / push は従来どおりユーザーが求めたときに行う。
- このルールは「どのブランチで作業するか」のみを定める。
  コミット粒度・push タイミングは既存ルール（`sd-safe-commit.md` 等）に従う。
