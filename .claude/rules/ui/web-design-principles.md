# Web UIデザイン原則

## 位置づけ

> **Work Firstの後に適用する品質基準。**
> まず動かす → ブラウザで確認 → **このルールで品質を上げる**。
> 動いていないUIにデザイン原則を適用しても意味がない。

## 8原則

### 1. 構図から始める

パーツの寄せ集めではなく、画面全体の「構図」から設計を開始する。
最初の画面は文書ではなく、1枚の完成された「ポスター」として成立させる。

### 2. 視覚階層を明確に

情報の優先度をサイズ・ウェイト・コントラストで表現する。

| 要素 | 役割 | 例 |
|------|------|-----|
| h1 | ページの主題 | 大きく、太く |
| h2 | セクション見出し | h1より小さく |
| 本文 | 説明 | 標準サイズ |
| 補助テキスト | 注釈・ヘルプ | 小さく、薄い色 |

### 3. 引き算で解決

装飾を足すのではなく、余白・配置・大きさ・コントラストで問題を解決する。

| 課題 | 悪い解決 | 良い解決 |
|------|---------|---------|
| 目立たせたい | 色を増やす、影を足す | サイズを大きくする、余白を広げる |
| 区切りたい | 線を引く | 余白を広げる |
| 強調したい | 太字+色+下線 | 太字のみ、またはサイズ変更 |

### 4. 制限による一貫性

| 要素 | 制限 |
|------|------|
| フォント | 最大2種類（見出し用+本文用） |
| 差し色 | 原則1色（ブランドカラー） |
| セマンティックカラー | 4色固定: success(緑), warning(黄), error(赤), info(青) |
| フォントサイズ | 3〜4段階のスケール |
| フォントウェイト | 3段階: normal(400), medium(500), bold(700) |

### 5. スペーシングの統一

8pxグリッドを基本とし、余白を一貫させる。

```
4px  (--space-1)  — 密接な要素間
8px  (--space-2)  — 関連要素間
12px (--space-3)  — 軽い分離
16px (--space-4)  — 標準の分離
24px (--space-6)  — セクション内の区切り
32px (--space-8)  — セクション間の分離
48px (--space-12) — 大きな区切り
```

### 6. 状態UIの完備

全ての動的コンテンツに以下の4状態を用意する:

| 状態 | 表示 |
|------|------|
| loading | スピナーまたはスケルトン |
| empty | 「データがありません」+ 次のアクション案内 |
| error | エラーメッセージ + リトライ手段 |
| success | データ表示（通常状態） |

### 7. 瞬読性の確保

- テキストは書いた後に30%削減する
- 見出しだけを読んで内容が理解できること
- 専門用語・デザイン用語を避け、その製品独自の言葉で語る

### 8. 1セクション1目的

- 1つの画面領域には1つの目的、1つの見出し
- 複数の目的を1セクションに詰め込まない

## 禁止事項

| 禁止 | 理由 | 代替 |
|------|------|------|
| カードUIの乱用 | 全部カードにすると情報の優先度が消える | リスト、テーブル、直接配置を検討 |
| ドロップシャドウ依存 | 安易な高級感演出、重なると汚い | 余白とボーダーで区切る |
| 色の氾濫 | 差し色が3色以上あると散漫になる | 差し色1色 + セマンティック4色 |
| 冗長テキスト | 読まれない長文は存在しないのと同じ | 30%削減ルール適用 |
| デフォルトフォント放置 | サービスの個性が消える | 意図的にフォントを選択する |
| 謎の太線・装飾アイコン | 視覚的ノイズ | 必要な情報のみ表示 |

## プロダクトUI（ダッシュボード等）向け補足

- ブランドの雰囲気より**使いやすさ・状態・次の操作**を優先する
- 見出しは「何があるか」ではなく「何ができるか」を記述する
- 例: "Selected KPIs" ではなく "Track your key metrics"

## マーケティングUI（LP等）向け補足

- 構成: ヒーロー(象徴) → サポート(機能説明) → ディテール(深掘り) → CTA(行動喚起)
- ヒーローの優先順位: ブランド名 > 見出し > 本文 > CTA > 画像
- ヒーロー画像は全幅（フルブリード）を基本とする
- ロゴを隠したときに他社サイトに見えるデザインは失敗

## GAS Web App 固有の追加制約

GAS Web Appはiframe内（`script.googleusercontent.com`）で動作するため、追加の制約がある:

| 制約 | 理由 | 対応 |
|------|------|------|
| 重いCSSアニメーション禁止 | iframe内のパフォーマンス制約 | transition: 0.2s 程度に抑える |
| `position: fixed` 禁止 | iframe内で期待通り動作しない | `position: sticky` または通常フロー |
| 外部フォント直接読込に注意 | CSP制約の可能性 | Google Fonts CDN推奨 |
| `window.location` 系API禁止 | iframe内で動作しない | `google.script.run` を使用 |

推奨スタック: **Tailwind CSS CDN**（GASではnpmビルドパイプラインが使えないため）

## デザイントークンとの関係

プロジェクトごとのカラー・フォント・余白は `.kiro/design/DESIGN_TOKENS.md` で定義する。
このルールは「原則」、デザイントークンは「具体値」という関係。

## 評価方法

UI実装後は `.claude/rules/ui/visual-review-checklist.md` のチェックリストで評価する。
合格ライン: 50/70以上。
