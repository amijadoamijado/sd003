# 開発哲学 - 2026年4月

## 起点

Claude Code開発者ボリスの2つの言葉。

> 「6ヶ月先を見てプロダクトを開発している。」

> 「汎用ツールの機能が上がれば、専用ツールは駆逐される。」

---

## 第1層：私を支える考え方（不変）

AIの進化は、開発の民主化を経て、開発そのものの消滅に向かう。

SaaSの時代は「作れる人」が強かった。
Claude Codeの時代は「作れなくても仕様を書ける人」が強くなった。
その先は「作る必要があるかどうかを判断できる人」が最も強くなる。

### 中間層の消滅

従来のシステム開発には、目的と実行の間に厚い中間層があった。

```
従来: 目的 → ロジック設計 → プログラミング → UI構築 → 操作学習 → 実行
これから: 目的 → 指示 → 実行
```

- **ロジック設計とプログラミング**（後面の中間層）: エンジニアが担っていた「思考をコードに翻訳する」仕事。AIが目的を受け取り、ロジックを自分で考え、自分で実行する。
- **UI構築と操作学習**（前面の中間層）: UI設計者が担っていた「コードを人間に翻訳する」仕事。AIが自然言語を理解するなら、翻訳装置としてのUIは不要になる。

これは極論に聞こえるが、AIが進化すれば必然。的確に指示すれば全て考えてやってくれるのだから。

### 学習コストの逆転

専用ツールは、人間にツールの作法を覚えることを強いる。ボタンの場所、画面遷移の順序、入力ルール。分からなければマニュアルを読み、詳しい人に聞き、サポートに電話する。学習の主体は常に人間側にある。

AIは構造が逆だ。分からないことはAIに聞けばいい。AI自身が壁に当たっても、自律的に調べ、試し、解決に向かう。

**専用ツール: 人間がツールに合わせて学ぶ。**
**AI: AIが人間に合わせて動く。**

### 意思決定の軸

6ヶ月先のLLMの進化を見越して、今のハーネスとスキルの設計に組み込む。

LLMは進化し続けている。この事実から2つの判断基準が生まれる。

1. **今は未成熟でも、6ヶ月後に完成する機能がある。** それを6ヶ月後に始めても遅い。だから今の設計に先行して組み込む。SD003のハーネス・スキル・Multi-AI協調基盤は、この先行投資そのもの。

2. **今ハーネスが担っている機能が、6ヶ月後にはLLM自体の能力になる可能性がある。** そのとき、その機能は不要になる。だから過剰に作り込まず、LLMの進化で不要になる部分は軽く作る。

つまり「専用ツールに投資するな」ではなく、「LLMの進化曲線の上に自分の設計を乗せろ」ということ。進化を恐れるのではなく、進化を前提にして設計する。AIの進化に賭ける側に立つ。

### 体験からの気づき

自分でツールを作ってから気づいた。画面設計がなくても目的を達成できる。
ボリスの言葉は「新しい発見」ではなく、「自分がやっていたことの名前がついた」感覚。

### 思想が先、実装が後

Multi-AI協調やハーネスによるAI統制の思想は、実装手段より先にあった。
各AIがバラバラで統一的に動かす手段がなかった時期から、揃う前提で設計していた。

そしてOpenAI Codex CLI、Anthropic Claude Code、Google Gemini CLIが公式に揃ったことで、実装が可能になった。SD003はその上にハーネス・スキル・協調基盤を構築した。

```
思想（先行）→ 設計（先行）→ 公式ツール登場 → 実装（SD003）
```

6ヶ月後に始めても遅い、の実例がこれ。公式が出てから設計を始めていたら、今の基盤はまだ存在していない。

### 共通器の設計思想

SD003のフレームワーク設計そのものも、この哲学の産物。

Claude、Gemini、Codex — どのAIでも同じ仕事ができる共通の「器」を作っている。コマンド正本は`.claude/commands/`に一元化し、`sync-cli-commands.py`でGemini TOML・Codex skillsに自動変換。`.handoff/RULES.md`が全AI共通のルール。

**なぜ共通器か。誰もがピンチヒッターになれるためだ。**

特定のAIに依存すると、そのサービスがダウンしたとき仕事が止まる。サーバー障害、API制限、料金体系の変更。どれも起きうる。共通器があれば、Claudeが落ちたらGeminiが、Geminiが落ちたらCodexが、同じフレームワーク上で同じ仕事を引き継げる。

```
器（共通）: SD003フレームワーク（ルール、コマンド、スキル、ワークフロー）
操作者（交換可能）: Claude Code / Gemini CLI / Codex CLI
```

これはAI協調でもあり、耐障害設計でもある。どのAIにも依存しない。フレームワークが仕事の質を担保し、AIはその上で動く実行者に過ぎない。

## 第2層：株式会社Sukima Digitalの考え方（半不変）

日本の中小企業には、デジタルの手が届いていない隙間が大量に残っている。

紙の台帳、Excelの手入力、電話とFAXの往復。あるいはSaaSを導入したものの、ツールの型に業務を押し込め、妥協したまま使い続けている現場。どちらも、業務のあり方そのものは一度も問い直されていない。

AIの進化は、この状況を一気に変える。汎用AIは専用ツールの機能を飲み込み、開発のハードルを消し、やがて開発そのものが不要になる領域を広げていく。SaaSを飛び越えて、アナログな現場がいきなりAIネイティブな業務に移行できる。

供給側から見ても、需要側から見ても、専用ツールの時代は終わりに向かっている。

ただし、AIは万能な道具であって、万能な設計者ではない。どの業務にAIを当て、どこに人間を残すか。そもそもその業務は存在すべきか。これを判断できるのは、現場の業務を骨まで理解し、全体を俯瞰して再設計できる人間だけだ。

**Sukima Digitalは、その設計者としてDXのラストワンマイルを埋める。**

デジタル化の残り作業を代行するのではない。業務のあり方そのものを問い直し、AIの能力を前提にゼロから組み直す。100社あれば100通りの最適解がある。それを一社ずつ設計する。

### ブランド階層

```
HUMONAIZ           ← 業務再設計の思想（上位概念）
  └─ AI-Bow        ← 実務支援ブランド
       ├─ Plan     ← 計画書・提案書・構想整理
       └─ Desk     ← バックオフィス支援
```

**AI-Bow（Bowの4つの意味）:**
1. **Bow = 礼儀（お辞儀）**: 顧客の業務に敬意を持って向き合う
2. **Bow = 船首**: 進むべき方向を示す
3. **Bow = 弓（バイオリン）**: その会社固有の音を奏でる
4. **Bow = 弓（武器）**: 結果として顧客の競争力になる

### サービスライン

| サービス | 対象 | 売るもの | スケール |
|---------|------|---------|---------|
| AI-Bow Plan | 会計事務所・中小企業 | 業務再設計力 | あみおの稼働依存 |
| AI-Bow Desk | 一人親方・美容師・整体師・飲食店 | バックオフィスのAI労働力 | パターン化で横展開可 |

### やらないことの定義

- 専用ツールの開発・販売（6ヶ月で陳腐化するものに投資しない）

- Carryflowerの税理士としての信頼

## SDの二重意味

SD003の「SD」はもともと **Spec Driven**（仕様駆動）の略。
コードより先に仕様を定義し、仕様が開発を駆動するという思想。

この考え方は、Sukima Digitalの事業哲学と本質的に同じ構造を持つ。

| SD | 意味 | 核心 |
|----|------|------|
| Spec Driven | 仕様が開発を駆動する | 「何を作るか」を先に定義する |
| Sukima Digital | 業務設計がAI活用を駆動する | 「何をさせるか」を先に定義する |

どちらも「目的の定義が先、実行は後」という同じ原則に立っている。
