# Grillme v1.0

改善フェーズで、AIの解決提案が難しすぎて人間が理解できない瞬間に、
AI自身が自分の回答を執拗に分解・再提示するスキル。

## このスキルが生まれた背景

- AIとチャットで課題解決を進める途中、AIの回答を人間が理解できない瞬間がある
- 理解できないまま「分かったふり」で承認すると、後で「思っていたものと微妙に違う」が起きる
- ただし人間に意図を書かせる設計は本末転倒（Intent-Diff Gate が却下された理由）。AIの良さを活かしていない
- AIには「優しい問題を難しく回答する癖」「本質ではなく枝葉を直す癖」がある

これらをAI側で吸収する。人間は「分かった」と宣言するだけで済む設計。

## 目的（最重要）

**改善（1→N）フェーズで、課題やエラー発生時、AIの解決提案が複雑すぎて人間が理解できないことを防ぐ「すり合わせ」。**

人間に書かせず、AI側が自分の解決提案を執拗に分解・再提示する。

## 適用基準

- **改善（1→N）フェーズで使う**（0→1 の要件定義は `/blueprint-gate`）
- 課題・エラー解決過程で AI解決提案が出ているが、人間が「分からない」と感じたとき
- 人間が起動を宣言したときのみ発火（自動判定なし）

すぐ理解できる回答に対しては起動しない。AIの回答に対して人間側が「これ難しすぎる」「本当にそれが本質か」と感じた瞬間に呼ぶ。

## 名称由来

Matt Pocock の grill-me（前向き：実装**前**にAIが人間を問い詰めて Context.md / ADR を生成し考慮漏れを排除する）の後ろ向き派生。

| 方向 | スキル | 対象 |
|------|--------|------|
| 前向き grill（原典） | SD003 では `/blueprint-gate` | 実装前の人間意図の引き出し |
| 後ろ向き grill | **`/grillme`（本スキル）** | 既出 AI 解決提案の分解 |

執拗に分解する動作実態に名称が一致するため Grillme に確定（議論履歴で Unpack / Essence Grill / Simplicity Anchor 等は却下）。

## 他スキルとの関係（フェーズ境界）

| フェーズ | 状況 | スキル |
|----------|------|--------|
| 0→1 | ゼロから何を作るか定義 | `/blueprint-gate`（原典 grill-me 相当） |
| 1→N | バグの真因追及 | `/bug-quick` または `/bug-trace` |
| 1→N | AIが同じエラーで迷走 | `/dialogue-resolution` |
| **1→N** | **AI解決提案が複雑すぎて理解できない** | **`/grillme`（本スキル）** |

`/grillme` は他スキルへの干渉なし。呼ばれていないときは存在しない（常駐しない）。終了後は元のチャット文脈に戻る。

## 動作フロー（3ステップ）

### Step 1: AI自問（人間に書かせない）

直前の AI 解決提案に対して、AI 自身が以下を自問する:

- この回答のうち、人間が分からない可能性がある箇所はどこか
- 自分が暗黙の前提として置いていることは何か
- 専門用語・論理飛躍・抽象概念がどこにあるか

人間には何も質問しない。

### Step 2: 構造化して再提示（HTMLアーティファクト）

以下の構造で再提示する:

| 要素 | 内容 |
|------|------|
| 結論 | 一行で言い切る |
| 根拠 | 階層構造（結論→なぜ→なぜなぜ） |
| 根拠の根拠 | さらに一段深い理由 |
| 暗黙の前提 | 独立ブロックで分離（AIが置いていた前提） |
| 用語定義 | 必要時のみ |
| 図解 | 必要時のみ。不要なら入れない |

出力は HTML アーティファクト（後述のフォーマット節）。
テンプレート: `assets/grillme-template.html`

### Step 3: 人間の応答パターンに対応

人間が指定する応答パターンは A / B の 2 種類。

**パターンA：人間が「ここが分からない」と特定できる場合**

- その箇所をさらに分解
- 例え話・対比・ビフォーアフター・別角度から繰り返し試す
- **一度の説明で諦めない**。執拗に分解する
- 人間が「分かった」と宣言するまで終わらない

**パターンB：人間が「どこが分からないか分からない」場合**

`AskUserQuestion` で「分からなさの種類」を最大 3 択提示する。人間に書かせない。選ぶだけで進む設計。

- 結論そのものが理解できない
- 結論は分かるが、なぜそうなるかが分からない
- 使われている言葉の意味が分からない

選択肢に応じて分解の角度を切り替える。

## 終了条件

- 人間が「分かった」「OK」「進めてよい」など明示的に宣言したときのみ終了
- **AI から「理解しましたか」とは聞かない**（AI 起点の終了確認は禁止）
- 判定権は完全に人間側

## 出力フォーマット（HTML）

| 項目 | 規定 |
|------|------|
| モード | ライトモード基調 |
| 背景 | 白（`#ffffff`） |
| 本文色 | 濃グレー（`#2c3338`） |
| アクセント | ネイビー / 青系を控えめに（`#2c5282` 等） |
| 前提ブロック背景 | 薄いネイビー（`#f0f4f8`） |
| 禁止配色 | 蛍光色・派手な配色 |
| 軽量原則 | スマホでも見渡せる量、スクロール最小 |
| 印刷対応 | `@media print` 対応、PDF 化に耐える |

クライアント説明への転用を想定するため、印刷・PDF 化に耐える設計とする。

テンプレート実体: `.claude/skills/grillme/assets/grillme-template.html`

## 図解・表の使い分け判断

| 内容 | 形式 |
|------|------|
| 複数の用語や選択肢を対比したい | 表 |
| 階層構造（結論→根拠→根拠の根拠） | ネスト表示またはツリー図 |
| 時間軸・手順 | フロー図 |
| 因果関係 | 矢印付き図 |
| ビフォーアフター | 2 カラム対比表 |
| 抽象概念 | 具体物に置き換えた図 |
| どれも該当しない | 図解は入れない（無理に入れない） |

## 設計思想の継承

あみおさんが新人研修体系に組み込んでいる「なぜ？ワーク」「プロセスワーク」と地続き。

| 研修ワーク | 本スキルでの対応 |
|-----------|------------------|
| なぜ？ワーク | 表層理由から本質理由への階層深掘り（根拠の根拠） |
| プロセスワーク | 結論ではなく根拠の連鎖を可視化 |
| Grillme | AI の回答に同じ思想を適用し、暗黙前提を露出させる |

組織導入時の認知コストを下げるため、研修語彙と地続きに設計してある。

## 議論履歴で却下された案（再提案禁止）

| 案 | 却下理由 |
|----|---------|
| Intent-Diff Gate | 人間に意図を先に書かせる設計 → 人間負荷過剰 |
| Refactor Grill | 既存改修用の影響範囲グリル → 別問題 |
| Simplicity Anchor | シンプル化を目的化 → 本質理解が目的、シンプル化は副産物 |
| Essence Grill | 本質擦り合わせの前向き設計 → 主体と方向が違う（本件は後ろ向き） |
| Unpack | 穏やかな解きほぐしのニュアンス → 動作実態（執拗に問い詰める）と不一致 |
