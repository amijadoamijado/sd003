# Jobs Review v1.0

「スティーブ・ジョブズなら満足するか？」を問いの軸に据え、UI/UX・操作性・対話体験・
成果物の完成度から改善点を引き出すメタ批評スキル。

## このスキルが生まれた背景

- UI/UXや、CLI・エージェントとの対話の「気持ちよさ」は定量チェックリストでは拾いにくい
- AIは自分の成果物に対して評価が甘くなりがちで、「動いているから完了」で止まりやすい
- 「ジョブズなら」という強い基準を仮に立てることで、普段素通りしがちな問いを強制的に立てさせる
- 対象は画面UIだけでなく、CLI/エージェントとの会話のテンポ・言葉遣い・確認の頻度、
  そして成果物そのものの完成度（このまま出して恥ずかしくないか）も含む

## 目的（最重要）

架空のジョブズを審判にすることが目的ではない。「これは本当に削れないか」「一気通貫の体験に
なっているか」「このまま出して恥ずかしくないか」という普段素通りしがちな問いを強制的に立てさせ、
**具体的な改善案を最低1つ引き出す**ための踏み台。感想や賞賛で終わらせない。

## 適用範囲（重要 — フロントに限定しない）

| 対象 | 例 |
|------|-----|
| 画面UI | Webアプリ、ダッシュボード、フォーム |
| CLI/エージェント対話 | Claude Code, Codex CLI, codex.appとのやり取りのテンポ・確認回数・言葉遣い |
| ドキュメント/レポート | 要件定義書、HTML成果物、レビューレポートの構成 |
| 成果物全体の完成度 | コード・機能・パッケージが「出荷可能な仕上がりか」 |

## 適用基準

- ユーザーが明示的に呼んだ時のみ発火（自動判定なし、grillmeと同様）
- UIレビューの仕上げ段階、または「完成しました」と報告する直前に使うと効果的（＝完成度判定モード）

## Jobsレンズ（批評軸）

| 軸 | 問い |
|----|------|
| シンプルさ | まだ削れる要素はないか |
| フォーカス | 何を捨てたか言えるか（Noと言えているか） |
| 一気通貫の体験 | 最初のタッチから最後まで一人の意思が貫かれているか、継ぎ接ぎ感はないか |
| 見えない部分への配慮 | エラー時・使われない分岐にも同じ丁寧さがあるか |
| 摩擦 | 不要な確認・クリック・待ち時間・冗長な説明はないか |
| 言葉づかい | システムの都合ではなく人間の言葉になっているか |
| 驚きと感情 | 機能を満たすだけでなく「おっ」と思う瞬間があるか |
| **完成度（出荷可能性）** | 今日これを世に出して恥ずかしくないか。仕上げ切っているか、それとも荒いか |

## 実行フロー（対話型）

### Step 1: 評価対象の特定
何を見るかを確認する（画面/操作ログ/文書/成果物一式）。ユーザーが渡していなければ確認する。

### Step 2: レンズごとの判定
各軸について「ジョブズなら満足するか」をYes/微妙/Noで一つずつ判定し、一言理由を添える（表形式）。

### Step 3: ブラッシュアップ案の抽出（核心）
微妙/No判定の軸について、具体的な改善案を**最低1つ**、可能なら2〜3案を対比で提示する。
抽象的な「もっと良くできる」で終わらせない。

### Step 4: 優先順位づけ
インパクト大×労力小のものから並べる。

### Step 5: 総合判定
一言で総合判定を述べ、「次にやるべきこと1つだけ」を明確に指す（一度に一つに絞るのもジョブズ的）。

## 出力フォーマット

通常はチャット内で表＋箇条書き。UI画面の視覚比較が有効な場合のみ、既存の `html-report` スキル
（`.claude/skills/html-report/`）を呼び出しビフォーアフター比較HTMLを生成する。新規テンプレートは作らない。

## 禁止事項

| 禁止 | 理由 |
|------|------|
| ジョブズの実在の発言・逸話を捏造して権威付けに使う | 架空引用は誠実性違反 |
| 「ジョブズならこう言う」を絶対的正解として押し付ける | 最終判断はユーザー。あくまで踏み台 |
| 改善案を伴わない「もっと良くできる」で終わらせる | 引き出す機能が働いていない（本スキルの目的違反） |
| 全項目Yesで終わらせて指摘ゼロにする | 甘い評価を防ぐのが目的。既に十分な場合はその理由を明示した上で完成度軸の粗探しに移る |
