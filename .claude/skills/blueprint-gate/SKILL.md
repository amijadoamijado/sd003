# Blueprint Gate v1.0

開発・制作に入る前に、あみおの思考をゴールから逆順で引き出し、正攻法の要件定義書として整形するスキル。

## このスキルが生まれた背景

- Opus 4.6の出力を信頼しすぎて前提確認を怠り、修正の連続になった
- Codexレビューがあるからと安心し、自分の判断介入が遅れた
- ゴールとアウトプットを定義せずにAIを走らせて時間を溶かした
- 他人のリポジトリを参考にした設計プロセスが自分の感覚に合わなかった

## 適用基準

以下のいずれかに該当する場合、Blueprint Gate必須:
- **タスクが1時間以上かかりそう**
- **ゴールとアウトプットが自分の頭の中で明確に言語化できない**

すぐ動くとわかっているもの、30分以内で終わるものは直接着手してよい（Work First）。

## Work Firstとの関係

**衝突しない。むしろ直結する。**

Work First = 「動かないものを作る行為を全て排除せよ」という上位概念。
ゴール未定義で走り出して修正の嵐になるのは Work First違反。
Blueprint Gateは「動くものを作るために、何が動く状態かを先に定義する」行為。

## 核心原理

AIは書かれていないことを存在しないものとして扱う。
だから人間の意図をテキストとして外部化する。
テキストに出すからこそ検証可能になり、AIの思い込みが入り込む余地が減る。

## 入力と出力の分離

あみおが考える順番（入力プロセス）:
1. ゴール → 2. アウトプット → 3. 検証観点 → 4. 背景 → 5. 現状と前提

読み手が読む順番（出力フォーマット）:
1. 背景 → 2. 現状と前提 → 3. ゴール → 4. アウトプット → 5. 要件 → 6. 検証観点

AIはこの変換を担う。あみおの入力順序を尊重し、出力時に正攻法へ組み替える。

## 対話プロセス（5フェーズ）

**一問ずつ聞く。まとめて聞かない。** 各フェーズ完了後にAskUserQuestionで確認してから次へ。

### Phase 1: ゴールの引き出し

> Q1「このプロジェクトが完了したとき、何がどうなっていれば成功か？一文で。」

ここで得た回答が要件定義書全体のアンカーになる。
曖昧な表現（「改善する」「効率化する」「良くする」）が出たら、具体的な状態に言い換えるよう促す。

### Phase 2: アウトプットの定義

> Q2「成果物は具体的に何か？」

以下を明確にする:
- 形式（ファイル種類、コード、ドキュメント、仕組み）
- 含まれるべき要素
- 使う人
- 使われるタイミングと場面
- 完成と未完成の境界線

### Phase 3: 検証観点の事前定義

> Q3「成果物が出てきたとき、何を見て合格とするか？」

**このフェーズが最重要。** アウトプットを定義した直後、思考がシャープなうちに検証観点を出す。
ゴール（Phase 1）から逆算して、具体的なチェック項目を列挙する。

ここが事前に定義されていることで:
- AIのセルフレビューが表面で止まらなくなる
- Codex等の外部レビューに観点を渡せる
- あみお自身のレビューも構造化される

### Phase 4: 背景の言語化

> Q4「なぜこれを作ろうと思ったのか？何に困っていたか？」

ここで出てくる「痛み」が、開発中に方向がずれたとき原点に戻るアンカーになる。

### Phase 5: 現状と前提の外部化

> Q5「今どうなっていて、何が使えて、何が制約か？」

以下を洗い出す:
- 技術環境（使用ツール、既存システム）
- 既にあるもの（流用可能なコード、設計、ドキュメント）
- スコープの境界線（やること / やらないこと）
- AIが暗黙に持ちそうな誤った前提（「こう解釈するな」リスト）

特に「AIが暗黙に持ちそうな誤った前提」は意識的に引き出す。

## 出力テンプレート（要件定義書）

5フェーズ完了後、以下の構成でマークダウンファイルを生成する。
保存先はあみおに確認する。

```markdown
# [プロジェクト名] 要件定義書

作成日: YYYY-MM-DD

## 1. 背景

なぜこのプロジェクトが必要か。どんな痛みがあるか。

## 2. 現状と前提

### 2.1 現状
今どうなっているか。

### 2.2 技術環境と制約
使えるもの、使えないもの。

### 2.3 スコープ
やること / やらないこと を明示。

### 2.4 AIへの注意事項
「こう解釈するな」リスト。暗黙の前提を潰す。

## 3. ゴール

一文で。このプロジェクトが完了したとき、何がどうなっていれば成功か。

## 4. アウトプット定義

### 4.1 成果物
何を、どの形式で作るか。

### 4.2 完成条件
どの状態になれば完成か。

### 4.3 利用者と利用場面
誰が、いつ、どう使うか。

## 5. 要件

### 5.1 機能要件
成果物が満たすべき機能。

### 5.2 非機能要件
パフォーマンス、保守性、拡張性など該当するもの。

## 6. 検証観点

成果物レビュー時のチェック項目。ゴールから逆算。

- [ ] 検証項目1
- [ ] 検証項目2
- [ ] 検証項目3
```

## 承認プロセス

**なし。** 要件定義書は開発中に方向がずれたとき原点に戻るためのもの。
承認ステータス管理は不要。動くものができたかどうかが最終判定。

## 開発中の原点回帰

開発中に方向がずれたと感じたら、要件定義書の「ゴール」と「背景」に立ち戻る。
AIが提案する変更や追加機能が、ゴールに直結するかを確認する。
