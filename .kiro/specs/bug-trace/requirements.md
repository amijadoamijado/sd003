# Bug Trace 要件定義書

## 基本情報
- **機能名**: Bug Trace - 3-Agent Parallel Investigation
- **バージョン**: 2.4.0
- **ステータス**: 実装済み
- **作成日**: 2026-01-02
- **最終更新**: 2026-01-03

## 1. 概要

### 1.1 目的
複雑なバグ調査を3つの専門エージェントで並列実行し、Ultrathink統合分析とビジュアルフロー図で根本原因を特定する。

### 1.2 背景
- 複雑なバグは単一視点では見落としが発生しやすい
- 仕様・実装・解決策の3観点から同時調査することで精度向上
- ビジュアル化により問題箇所の理解を促進

### 1.3 成功基準
- 3エージェント並列実行による調査効率化
- Ultrathink統合による高精度な根本原因特定
- ASCII図による視覚的な問題理解
- 日本語解説による非エンジニアへの説明力

## 2. 機能要件

### REQ-BT-001: 過去バグ検索
- **概要**: 調査前に過去の類似バグを検索
- **入力**: エラー内容、キーワード
- **処理**: BUG_INDEX.md検索、関連バグ特定
- **出力**: 関連バグリスト（Bug ID、タイトル、関連度）
- **優先度**: Medium

### REQ-BT-002: 3エージェント並列調査
- **概要**: Spec/Code/Solutionエージェントの並列実行
- **詳細**:
  - **Spec Agent**: 仕様書調査（要件、設計、期待動作）
  - **Code Agent**: コード追跡（エントリポイントからエラー箇所まで）
  - **Solution Agent**: 仮説立案（原因候補、証拠、修正アプローチ）
- **入力**: エラー情報、再現手順
- **処理**: Task toolで3エージェント並列起動
- **出力**: 各エージェントの調査報告
- **優先度**: High

### REQ-BT-003: Ultrathink統合分析
- **概要**: 3エージェント結果の深層統合分析
- **詳細**:
  - エージェント間の相互参照
  - 矛盾・ギャップの特定
  - 仮説の優先順位付け
- **入力**: 3エージェントの報告
- **処理**: 拡張思考による統合分析
- **出力**: 統合分析レポート、日本語解説（必須）
- **優先度**: High

### REQ-BT-004: ビジュアルフロー図生成
- **概要**: ASCII artによるエラーフロー可視化
- **詳細**:
  - Error Flow Diagram（入力→処理→エラー）
  - Reverse Trace（エラー→根本原因への逆追跡）
  - Divergence Analysis（期待vs実際の比較）
- **入力**: 調査結果
- **処理**: ASCII図生成
- **出力**: 3種類のASCII図 + 各図の日本語解説（必須）
- **優先度**: High

### REQ-BT-005: 仮説形成・検証
- **概要**: 根本原因の仮説を優先順位付きで提示
- **入力**: 統合分析結果
- **処理**: 仮説ランキング、証拠整理
- **出力**: 仮説テーブル（順位、仮説、証拠、信頼度）
- **優先度**: High

### REQ-BT-006: 解決策提案・実行
- **概要**: 修正計画の策定と実行
- **詳細**:
  - 修正対象ファイル特定
  - リスク評価
  - テスト計画
  - ロールバック計画
- **入力**: 承認された仮説
- **処理**: 修正計画策定、ユーザー承認後実行
- **出力**: 修正完了、テスト結果
- **優先度**: High

### REQ-BT-007: バグレポート生成
- **概要**: 調査結果を番号付きレポートとして保存
- **詳細**:
  - Bug ID自動採番（BUG-{PROJECT}-{SEQ}-{TIMESTAMP}）
  - レポート保存（docs/troubleshooting/bug-reports/）
  - インデックス更新（BUG_INDEX.md）
- **入力**: 全調査結果
- **処理**: レポート生成、インデックス更新
- **出力**: バグレポートファイル
- **優先度**: Medium

### REQ-BT-008: 乖離分類（Divergence Classification）
- **概要**: 調査結果を分類し、適切な対応アクションを決定
- **詳細**:
  - **Code Defect**: 仕様が最新かつ明確で、コードが逸脱 → コード修正へ
  - **Spec Defect**: コードの動作が合理的で、仕様が古い/曖昧 → 仕様書更新へ
  - **Both Defect**: 両方に問題あり → 優先順位付けて両方対応
  - **Ambiguous**: 判断困難 → ユーザー確認
- **判定基準**:
  - 仕様書の最終更新日 vs コードの最終変更日
  - コードの動作が「意図的改善」か「バグ」か
  - 他コードとの整合性
  - ビジネスロジックの妥当性
- **入力**: Ultrathink統合分析結果
- **出力**: 分類結果、推奨アクション
- **優先度**: High
- **バージョン**: v2.4.0で追加

### REQ-BT-009: 仕様書更新提案（Spec Update Proposal）
- **概要**: Spec Defect時に仕様書の更新提案を作成
- **詳細**:
  - 更新対象ファイル・セクションの特定
  - 現在の記述 vs 提案する記述の比較
  - 影響範囲の分析
- **入力**: 乖離分類結果（Spec Defect）
- **処理**: 仕様書更新提案作成、ユーザー承認後に更新
- **出力**: 更新された仕様書、変更ログ
- **優先度**: High
- **バージョン**: v2.4.0で追加

## 3. 非機能要件

### NFR-BT-001: ユーザー確認必須
- 各Step完了後にAskUserQuestionで確認
- 自動進行禁止
- 修正実行前に承認必須

### NFR-BT-002: 日本語解説必須
以下4箇所で日本語解説が必須：
- Ultrathink統合分析解説
- Error Flowフロー解説
- Reverse Trace逆追跡解説
- Divergence Analysis乖離分析解説

### NFR-BT-003: 出力フォーマット
- テーブル形式での差異報告
- ASCII図の標準テンプレート使用
- 日本語解説の配置ルール遵守

## 4. 禁止事項

1. ユーザー確認をスキップする
2. 自動で次のStepに進む
3. Step 7承認前に修正を開始する
4. エージェントの調査結果を無視する

## 5. 実行フロー

```
Step 1: エラー情報収集 + 過去バグ検索
    ↓ [AskUserQuestion]
Step 2: 3エージェント並列調査
    ↓ [AskUserQuestion]
Step 2.5: Ultrathink統合分析 + 乖離分類
    ↓ [AskUserQuestion]
    ├─ Code Defect → Step 3へ
    └─ Spec Defect → Step 2.6へ

[Code Defect の場合]
Step 3: ビジュアルフロー図生成
    ↓ [AskUserQuestion]
Step 4: 仮説形成
    ↓ [AskUserQuestion]
Step 5: 解決策提案
    ↓ [AskUserQuestion]
Step 6: 修正計画策定
    ↓ [AskUserQuestion]
Step 7: 修正実行
    ↓ [AskUserQuestion]
Step 8: バグレポート生成

[Spec Defect の場合]
Step 2.6: 仕様書更新提案
    ↓ [AskUserQuestion: 承認]
仕様書更新 → 完了
```
