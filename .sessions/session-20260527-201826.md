### 備考

#### `.sd/` 消失の真因（確定）

2層構造:
1. **gitignored-but-tracked の矛盾** が Claude Code runtime (#34330) の working tree refresh で「捨てて良いファイル」と誤判定される
2. **Edit/Write tool on `.sd/` + 次の Bash** が wipe trigger（settings.local.json の慢性 M も発火確率を上げる）

post-commit hook の auto-restore は完全消失時のみ発火（`[ ! -d ".sd" ]`）。partial wipe には対応不可。

#### 本セッションで確立した運用ルール

| 操作 | 方法 |
|------|------|
| `.sd/` ファイル作成 | Bash heredoc/redirect（`cat > file << EOF`） |
| `.sd/` ファイル編集 | Bash（`sed -i`, `echo >>`） |
| `.sd/` ファイル読み | Read tool ✅ |
| `.sd/` への Write/Edit/MultiEdit | 物理ブロック ⛔（PreToolUse hook） |
| `.sd/` 削除 | block-sd-destructive.sh で既ブロック |

#### at002 が示した教訓

at002 は意図せず Layer 1+2 が入っていた（`/sd-upgrade` 経由と推定）。**Layer 1+2 だけで実用上は十分** という実機証拠。Layer 3 は予防策。sd003 で今日まで未修正だったのは、`/sd-upgrade` を sd003 自身に適用していなかった可能性が高い。

#### post-commit hook の補強候補

partial wipe 検知（`[ ! -d ".sd" ]` ではなく、HEAD と working tree の `.sd/` ファイル数比較で発火）を追加すれば、Layer 3 ガードレールが破られた場合の最終防衛線になる。
